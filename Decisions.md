# Architectural Decisions

## Why Django Was Chosen

Django was selected because it provides:

- rapid backend development
- strong ORM support
- admin tooling
- authentication capabilities
- mature ecosystem
- PostgreSQL compatibility

---

## Why PostgreSQL Was Chosen

PostgreSQL was selected because:

- it is enterprise-grade
- supports JSON fields
- strong relational modeling
- widely used in SaaS applications
- excellent Django integration

---

## Why React Was Chosen

React was selected because:

- component-based architecture
- strong ecosystem
- modern frontend workflows
- reusable UI patterns
- excellent API integration

---

## Why CSV Was Chosen For SAP

CSV ingestion was selected because:

- realistic for enterprise operations
- manageable implementation scope
- demonstrates ingestion complexity

---

## Why CSV Was Chosen For Utility Data

Utility providers commonly expose:

- billing CSVs
- spreadsheet exports
- portal downloads

---

## Why JSON Was Chosen For Travel Data

Travel systems are commonly API-driven SaaS platforms.

JSON ingestion was selected because:

- realistic for travel APIs
- demonstrates ingestion diversity
- simulates cloud SaaS integrations

---

## Why Render Was Chosen For Deployment

Render was selected because:

- simple deployment workflow
- PostgreSQL integration
- Django compatibility
- frontend static hosting
- beginner-friendly infrastructure
